# 治理闭环规则

> 最后更新：2026-07-22
> 设计：DeepSeek + GLM 评审 + 先生确认

---

## 一、治理项生命周期

```
创建（Reflector/DeepSeek）
   ↓ pending
投递（推送通知给接收方）
   ↓ delivered
确认（接收方阅读）
   ↓ acknowledged
执行中（执行治理操作）
   ↓ executed / failed
完成或失败
```

### 状态说明

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| `pending` | 已创建，未投递 | 治理项产出时 |
| `delivered` | 已投递到接收方 | 推送通知成功时 |
| `acknowledged` | 接收方已阅读 | 接收方阅读时（manual/auto） |
| `executing` | 正在执行 | 开始执行操作时 |
| `executed` | 执行成功 | 操作完成无错误 |
| `failed` | 执行失败 | 操作出错或超时 |

### 重试策略

- 投递失败：最多重试 3 次，间隔 1min → 5min → 15min（指数退避）
- 执行失败：最多重试 2 次，间隔 5min → 30min
- 超过重试上限：自动升级到下一个优先级级别（🔴 级告警）
- 连续 7 天状态均为 `failed`：自动生成错误报告

## 二、白名单规则

详见 `governance/whitelist.json`，核心原则：

- **按操作类型划分**，而非按优先级标签
- 白名单内 🟢 操作 DeepSeek 自行执行
- 白名单外 🟢 操作暂缓，先生确认
- 禁止类操作即使标 🟢 也不得自执行
- 单日自执行上限 5 项

## 三、通知分发规则

```
┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│  Reflector  │ ──→ │   DeepSeek   │ ──→ │    先生      │
│  产出治理    │     │  分级处理    │     │  最终裁决    │
└─────────────┘     └──────────────┘     └──────────────┘
```

| 级别 | 投递目标 | 投递方式 | 格式 |
|:----:|---------|---------|------|
| 🔴 | DeepSeek → 先生（即时） | feishu 消息 | 单条高亮，含优先级标记 |
| 🟡 | DeepSeek（执行）+ 先生（每日） | feishu 每日摘要 | 列表汇总，不逐条推送 |
| 🟢 白名单内 | DeepSeek 自执行 | 无需推送 | 在每日摘要末尾记录 |
| 🟢 白名单外 | 暂缓，先生下次对话确认 | 对话起始时提醒 | 简短列表 |

## 四、异常恢复

### 管道断裂处理

1. **发现断裂**：任何节点检测到前序节点超时无产出（Reflector >24h 未产出、flush >36h 未执行）
2. **自动诊断**：检查 cron 运行状态、脚本是否在位、配置文件完整性
3. **自动恢复**：管理员状态重建
4. **通知先生**：若自动恢复失败，标注 🔴 推送给先生

### 堆积治理项批量处理

- 治理项堆积超过 10 项时自动触发批量清理（先生确认后执行）
- 优先级顺序处理：先 🔴 → 🟡 → 🟢
- 同类操作批量执行（例如一次性更新所有头部日期）

## 五、健康检查

每日自动检查（由 Reflector 日常反思附带）：

| 检查项 | 正常 | 异常 |
|--------|:----:|:----:|
| Reflector 日产出 | <24h | >24h ⇒ 告警 |
| memory/ 日日志 | <36h | >36h ⇒ 告警 |
| tracker 无过期 pending | <48h | >48h ⇒ 提示 |
| 治理项堆积数 | <10 | ≥10 ⇒ 批量清理 |
| cron 任务状态 | error=0 | error≥1 ⇒ 告警 |
