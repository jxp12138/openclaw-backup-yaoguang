# 瑶光记忆系统 — 最终架构设计

> 版本: Final v1.0
> 日期: 2026-07-18
> 基础: OpenClaw 2026.7.1 + memory-core 插件
> 三轮评审: 我与 GLM 达成一致

---

## 一、设计原则

### 原则 1：感知-动作反射弧

Agent 行为越接近"感知→动作"的短回路，越可靠。
避免：延迟执行、分心执行、多步编排。

*凡需要"过一会儿再做"的设计，都应该改成"现在就做"或"等触发信号明确再做"。*

### 原则 2：信号强度决定自动化程度

| 等级 | 信号类型 | 示例 | 可靠性 | 设计策略 |
|------|---------|------|--------|---------|
| S | 确定性事件 | 压缩、session_end、cron 触发 | 100% | 框架 Hook |
| A | 强语义信号 | "记住这个"、用户提出新话题 | ~90% | LLM 自觉 + AGENTS 指引 |
| B | 弱语义信号 | "大概第几轮了"、"会话快结束了" | ~10% | 不依赖，改执行模型 |
| C | 无信号 | 后台静默 review | 0% | 不做自动化 |

*只有 S 级和 A 级允许自动化或半自动化。B 级和 C 级来源的改动必须先验证需求再投入。*

### 原则 3：替代性满足检测

当发现"框架已内置 X"时，先检验：
> 框架的 X 的产出物，和我需要的 X 的产出物，一样吗？

*不同则不成立。*（案例：Auto Memory Flush 产出的是 LLM 摘要，不是原始对话记录）

### 原则 4：先生是判断者，不是执行者

所有设计：
- 降低先生需要做的事情数量
- 降低每件事的复杂度上限
- 先生只做"行/不行"的判断，不做脚本/配置/代码操作

## 二、架构图

```
┌─────────────────────────────────────────────────────────────┐
│                    用户输入（任何通道）                       │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  ① 新话题 / 需要上下文 → 手动 memory_search               │
│     信号：A 级（强语义）→ AGENTS 指引驱动                  │
│     效果：回复质量提升 + 上下文连续                        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼

│  ② "记住这个" / 我认为值得记 → 写入 long-term/            │
│     信号：A 级（强语义）→ 对话中即时执行                   │
│     工具：edit / write                                      │
│     产出：user-profile / feedback-log / project-context     │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  ③ Auto Memory Flush（框架层，自动）                       │
│     信号：S 级（压缩前确定性事件）                          │
│     触发：压缩前静默 LLM turn                               │
│     产出：memory/YYYY-MM-DD.md（LLM 摘要）                 │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  ④ Dreaming Light → REM（框架层，自动）                    │

│     产出：DREAMS.md（Light 暂存 → REM 主题反思）          │

└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  ⑤ Cron 每日 git 备份（框架层，自动）                      │
│     信号：S 级（cron 确定性）                               │
│     效果：灾难恢复                                          │
└─────────────────────────────────────────────────────────────┘
```

## 三、各层职责与配置

### 3.1 memory_search（已生效）

| 项目 | 说明 |
|------|------|
| 提供者 | memory-core 插件（已启用） |
| 触发 | 人工（我收到新话题或需参考上下文时） |
| 搜索范围 | MEMORY.md + memory/ + long-term/ |
| 后端 | SQLite FTS5 trigram（中文分词） |

### 3.2 手动写入 long-term/（已稳定运行）

| 项目 | 说明 |
|------|------|
| 触发条件 | 先生明确要求"记住" / 我认为值得记 |
| 工具 | edit / write |
| 写入目标 | long-term/{user-profile,feedback-log,project-context,references}.md |
| 冲突检测 | 写入前用 memory_search 检查是否已存在 |

### 3.3 Auto Memory Flush（已启用，默认生效）

| 项目 | 说明 |
|------|------|
| 配置 | `agents.defaults.compaction.memoryFlush.enabled`（默认 true） |
| 触发时机 | 上下文压缩前 |
| 产出物 | LLM 挑选的重要上下文 → `memory/YYYY-MM-DD.md` |
| 注意 | 产出的是**摘要**，不是原始对话记录 |

### 3.4 Dreaming（待开启）

| 项目 | 说明 |
|------|------|
| 配置路径 | `plugins.entries.memory-core.config.dreaming` |
| 开启阶段 | Light + REM（先不开 Deep） |
| 验证窗口 | 开启后 24 小时检查 DREAMS.md |
| 决策 | Light 排序暂存 → REM 主题反思 → 先生审阅 DREAMS.md → 确认提升 |
| Deep 阶段 | 等 Light+REM 运行 2 周后，验证事实提取质量再决定 |

### 3.5 Cron 每日 git 备份（已启用，保留）

| 项目 | 说明 |
|------|------|
| 触发 | cron 0 23 * * * |
| 效果 | 工作区每日自动推送到 git |
| 状态 | 不依赖 LLM，从未失效 |

## 四、不做的事情

| 事项 | 原因 | 触发条件 |
|------|------|---------|
| 不写 session_end hook 插件 | 需求未验证 | 如果 2 周内先生需要精确检索原文 |
| 不写 MCP Server | 当前场景不需要 | 如果后续需要外部知识库接入 |
| 不开 Deep Dreaming | 中文事实提取质量待验证 | Light+REM 运行 2 周后评估 |
| 不开 Active Memory | 怕延迟 + 手动指引足够 | 一周内遗漏 3 次以上 memory_search |
| 不开新 shell 脚本 | 框架能力已覆盖 | 先生明确要求某能力缺失 |

## 五、一周后复盘检查清单

| # | 检查项 | 通过标准 | 不通过怎么办 |
|---|--------|---------|------------|
| 1 | DREAMS.md 有内容吗？ | 有至少 1 条 Light 或 REM 记录 | 检查 dreaming 日志 > 查调度 |
| 2 | memory/YYYY-MM-DD.md 自动写入了吗？ | 有至少 1 次压缩记录 | 检查 compaction.memoryFlush 配置 |
| 3 | 出现过"应该搜但忘搜"的情况吗？ | 没有或仅 1 次 | 3 次以上 → 开 Active Memory |
| 4 | 需要查原始对话记录吗？ | 没有或请求可被摘要满足 | 需要精确原文 → 考虑插件 |
| 5 | long-term/ 手动写入正常吗？ | 有写入记录 | 检查触发习惯 |
| 6 | cron 备份正常吗？ | 有成功日志 | 检查 cron 状态 |
| 7 | 整体感觉如何？ | 正常 / 有改进空间 | 调整配置或方案 |
