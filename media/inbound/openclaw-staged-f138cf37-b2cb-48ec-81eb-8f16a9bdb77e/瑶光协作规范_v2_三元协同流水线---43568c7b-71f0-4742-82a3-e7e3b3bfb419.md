# 瑶光协作规范 v2 — 三元协同流水线

> 版本：v2.0
> 生成时间：2026-07-20 19:42
> 状态：待部署
>
> ### 变更摘要
>
> **合并 C+ 模式与持久 Agent 架构**
>
> 1. **废弃临时验证子 Agent** — 不再 spawn 临时子 Agent + curl 硅基流动调 GLM API
> 2. **统一为持久 GLM Agent 审查** — 通过 handoff/ 文件名状态机异步审查
> 3. **审查链路简化** — 硅基流动中转层移除，审查结果自然存入 handoff/ 供 Reflector 读取
> 4. **保留探索子 Agent 与落地验证子 Agent** — 这俩无冲突，继续使用 sessions_spawn

## 一、设计哲学（继承 C+ 模式）

| 原则 | 说明 |
|------|------|
| **从简单开始** | 不提前搭建复杂架构，当前够用就维持 |
| **硬约束兜底，软约束补缝** | 能用工具集隔离的不用 prompt 约束 |
| **不信任模型的自觉性** | 能用规则写清楚的绝不留给模型临场判断 |

## 三、三元协同流水线

### 3.1 标准流程

```
先生 → 主 Agent（我）：任务需求
  │
  ▼
[阶段 1：环境探索]（可选，🔴 红色任务前置）
  探索子 Agent → 环境盘点 → 附入审查指令包
  │
  ▼
[阶段 2：方案设计 + 审查指令包]
  我产出方案 + 审查指令包
  │
  ▼
[阶段 3：异步审查]
  我写 handoff/xxx-from-deepseek-to-glm.md
    → 先生切到 GLM 对话
    → GLM 读到 handoff（open 状态）
    → GLM 逐条审查 → 标记 .resolved.md
  │
  ▼
[阶段 4：决策]
  我读取 GLM 审查结果 → 完整转发先生
  先生决策：通过 / 有条件通过 / 不通过
  │
  ▼
[阶段 5：执行落地]（通过后）
  我按方案执行改动
  │
  ▼
[阶段 6：落地验证]（🔴 红色任务强制）
  落地验证子 Agent → 三对照核对
  │
  ▼
[阶段 7：事后反思]（异步，每日 03:00）
  Reflector 读 handoff/ + long-term/ → 输出反思报告 + 治理建议
```

### 3.2 与旧 C+ 模式的关键差异

| 维度 | 旧 C+ 模式 | 新三元协同流水线 |
|------|:---------:|:---------------:|
| 审查方式 | spawn 子 Agent + curl 硅基流动 | handoff/ 文件 → 持久 GLM Agent |
| 审查速度 | 同步（秒级） | 异步（需切换对话） |
| 依赖中转 | 硅基流动 API | 无中转，直连持久 GLM |
| 上下文连贯性 | 无（每次从零开始） | 有（GLM 可回顾历史 handoff） |
| 审查记录 | 子 Agent 上下文丢弃 | 自然存入 handoff/，Reflector 可读 |
| 探索子 Agent | 保留 | ✅ 保留 |
| 落地验证子 Agent | 保留 | ✅ 保留 |

### 3.3 审查结果直达先生（继承 C+ 模式）

- GLM 审查结果完整转发先生，不删减、不缩写、不选择性转述
- 我可在前面附摘要（标注"以下为摘要"），但不能替代原文
- 我可在后面附"我的回应"，标注不同意的条目及理由

**强制行为序列**：
> 收到 GLM 审查结果后，第一动作必须是：将原始内容完整粘贴到代码块中。粘贴完成之前不得生成任何分析、摘要或回应。

## 四、任务分级

| 级别 | 范围 | 验证路径 | 先生介入 |
|:----:|------|---------|:--------:|
| 🔴 **红色** | 方案设计/架构决策/安全配置/外部接入/涉及金钱/先生指定 | 探索→方案→GLM 审查→先生决策→落地→验证→反思 | 必须看结果并拍板 |
| 🟡 **黄色** | 代码实现/文档输出/信息整理/方案已定 | 方案→GLM 审查→先生确认 | 可秒阅确认 |
| 🟢 **绿色** | 简单查询/日常提醒/先生说"直接做" | 主 Agent 直接执行 | 无需介入 |

## 五、固定审查基线（6条）

| 编号 | 检查项 | 审查方法 |
|:----:|--------|---------|
| 1 | **需求覆盖** | 逐条对照原始需求，检查每条是否都有对应方案 |
| 2 | **内部一致性** | 检查方案各部分间是否存在逻辑冲突 |
| 3 | **可落地性** | 检查涉及的每项资源是否在现有环境中可用 |
| 4 | **异常处理** | 检查方案是否覆盖失败场景 |
| 5 | **过度设计** | 检查是否存在当前需求不需要的模块/步骤 |
| 6 | **假设透明** | 找出方案中所有隐含假设，评估风险 |

## 六、审查指令包规范

每次产出方案时同步生成审查指令包，写入 handoff/：

### 格式

```
# handoff/YYYY-MM-DD-HHmm-from-deepseek-to-glm.md

from: deepseek
to: glm
type: 方案审查请求
priority: 🔴/🟡/🟢
context: <简要背景说明>

## 固定基线（6条）
1. 需求覆盖 — "我要的事，方案全管到了吗？"
2. ...（逐条列出）

## 本次补充的专项验证点
1. ...
2. ...

## 已知风险（主 Agent 自查）
- ...
⚠️ 以上列表可能不完整，GLM 应独立判断。

## 设计假设
1. ...（若不成立，影响是...）
2. ...

## 待审查方案
（方案全文）
```

## 七、审查反馈闭环

每次审查走完后，主 Agent 主动问一次先生反馈：

> "先生，这次 GLM 审查有哪条查得好、哪条跑偏了？"

记录到 `review-feedback.json`：

```json
{
  "reviews": [{
    "task": "任务名称",
    "date": "2026-07-20",
    "rating": 4,
    "valuable": ["第3条异常处理查得准"],
    "false_positive": ["第2条认为过度设计，但实际必要"]
  }]
}
```

连续 3 次先生回"没问题"后，降级为月度汇总。

## 八、异常处理

| 异常情况 | 处理方式 |
|---------|---------|
| **GLM 对话未激活**（长时间无切换） | 我提示先生"GLM 有审查待处理，请切到 GLM 对话查看" |
| **GLM 输出格式不符合规范** | 按结构化模板手动整理，缺失信息标注"GLM 未提供" |
| **探索子 Agent spawn 失败** | 重试 1 次；仍失败 → 手动环境检查 |
| **落地验证子 Agent spawn 失败** | 重试 1 次；仍失败 → 我手动核对，标注"降级为手动验证" |
| **反馈积累 ≥3 条误报/遗漏** | 自动评估是否建立审查教训库（lessons.json） |
