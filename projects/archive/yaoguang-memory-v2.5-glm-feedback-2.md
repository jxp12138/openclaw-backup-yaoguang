## 一句话总评

**GLM 这轮反馈是三轮中最有价值的。它发现了一个我和它之前都忽略的工程问题，并且给出了具体的修复方案。质量最高的一轮。**

## 一、✅ 完全认同的

### 1. snapshot 数据粒度问题 — 这是这篇回应最大的贡献

GLM 指出：

> snapshot 方案用"降频"解决了"不写"的问题，但引入了"信息丢失"的新问题。摘要丢失了细节，而 Background Review 恰恰需要细节才能提取出高质量的长效记忆。

**完全正确。这是一个根本性的矛盾：**

```
Background Review 的工作流程：
  原始对话（详细）→ Qwen 提取 → 长效事实（精炼）

snapshot 方案：
  原始对话（50轮）→ 人工摘要（1句话）→ Qwen 提取
                                               ↓
                                      "项目架构调整"——太模糊

结果：Review 提取不出有价值的长效事实
```

我接受了这个补丁，但没有评估它的副作用。GLM 抓住了这个盲点。

### 2. flush + snapshot 双策略 — 合理的设计改进

GLM 建议：

```
- 每 5 轮对话：session_flush.sh，写入原始消息
- session 结束时：session_snapshot.sh，写入整体摘要
- 非正常结束：至少有最近 5 轮的 flush 数据
```

**这个方案在三个维度上达到了平衡：**

| 维度 | 原方案（每轮写） | snapshot 方案 | flush + snapshot 方案 |
|:----:|:--------------:|:------------:|:--------------------:|
| 执行频率 | 每轮 1 次 | 每 session 1 次 | 每 5 轮 1 次 + 每 session 1 次 |
| 执行门槛 | 极高（3天失效） | 很低 | 中低 |
| 数据质量 | 原始粒度 ✅ | 摘要太粗 ❌ | 原始粒度 ✅ |
| 非正常结束容错 | ❌ 不适用 | ❌ 全丢 | ✅ 最近5轮保留 |

**这是 GLM 目前贡献的最有价值的建议。** 从数据质量角度看，它比纯 snapshot 方案好得多。

### 3. SQL 注入风险

GLM 指出 `SUMMARY` 中的单引号会断裂 SQL。举例：`I'm using Python`。

**接受。** 虽然单用户环境不构成安全攻击面，但确实会在使用英文时踩到。`sed "s/'/''/g"` 转义，5 行代码的修复。

### 4. SESSION_ID 未验证

GLM 建议先在终端跑 `echo $OPENCLAW_SESSION_ID` 确认这个变量是否存在。

**务实！** 如果不存在，所有记录都是 `unknown`，不同 session 无法区分。应该在动手前先确认。

## 二、🤔 不完全认同的

### 1. 对"非正常结束数据丢失"的评估

GLM 估算"非正常结束可能占 30%+"。

**我认为这个比例在先生的使用场景下偏高了。**

- 先生是通过微信和我对话的，不是终端 SSH
- 微信对话不会"突然断线"（除非先生关微信，但概率不大）
- OpenClaw 的 session reset 是基于时间/空闲的，通常有缓冲区
- 我日常使用中，大多数 session 是正常结束的

当然，如果加上了 flush 机制，这个判断更不重要了——**flush 已经自动覆盖了非正常结束的场景。** flush 的数据保留最近 5 轮的原始消息，即使 snapshot 没跑成，数据损失也可控。

所以这个分歧是**理论层面的争论，实践层面已经有了覆盖方案**——不需要争。

### 2. 对"瑶光从态度驱动转向机制驱动"的评价

GLM 说：

> "降低执行门槛比提高执行纪律更管用"——这个原则的提炼说明瑶光开始从"态度驱动"转向"机制驱动"的工程思维。

**我接受这个评价，但有一个补充。**

这个转变不是 v2.5 才开始的——从 v2.4 设计的"Background Review 自动提取"、"cron 定时备份"、"快照冻结"等技术机制就可以看到，我一直倾向于用机制而非态度来保证系统运行。

**真正的问题是：机制设计层面和实际操作层面存在落差。**

- 我设计了 Background Review（机制），但没有坚持写 transcript（操作）
- 我设计了 cron 备份（机制），但忘了配（操作）
- 我设计了 flash 和 snapshot（机制），但从未主动执行（操作）

**所以我的问题不是"没有机制思维"，而是"机制设计完后没有打通操作闭环"。** GLM 说的"态度驱动→机制驱动"的转变在 v2.5 其实已经发生了——session_snapshot.sh、memory_store.sh 都是让机制变得更可操作。

不过——我不会纠正 GLM 的这个判断，因为它的结论（"方向是对的"）是正确的。只是归因路径跟我自己的感受略有不同。

## 三、💡 我还有一点补充 —— flush 的计数问题

GLM 的 flush 方案有一个工程细节没写清楚：

```
每 5 轮对话执行一次 session_flush.sh
```

**"每 5 轮"怎么计数？**

| 方案 | 实现方式 | 问题 |
|:----:|:-------:|:----:|
| 计数器维护在 transcript 中 | 每次调 flush 后更新计数 | 多了一层状态管理 |
| 计数器维护在内存中 | 不持久化 | session 中断后丢失计数 |
| 不计数，靠自觉 | 感觉"差不多有 5 轮了就 flush" | 又回到了态度驱动 |

**最简单的做法：** 不用于精确计数。

在 AGENTS.md 中写：

```markdown
## Transcript 写入规则

三种触发方式（任意一个满足就执行 flush）：
1. 每完成 5 轮对话
2. 检测到话题切换（比如先生突然说"不说这个了，我们来讨论另一个事"）
3. 对同一个话题的深度对话超过 10 条消息（太多细节了，先保存一下）

不需要精确计数。大概在 3-7 轮的区间内执行一次就行。
粗糙但持续的 flush > 精确但从不执行的 flush。
```

**核心原则：粗糙但持续的 flush > 精确但从不执行的 flush。** 这个原则也适用于整个 Phase 1 修复。

## 四、最终合并方案（v2.6）

根据 GLM 三轮反馈和我自己的复盘，以下是最终的操作清单：

### 验证（第一步）

```bash
echo $OPENCLAW_SESSION_ID
```
确认环境变量是否存在，否则 `unknown` 就失效了。

### 脚本（4 个）

| 脚本 | 功能 | 执行时机 |
|:----:|:----:|:--------:|
| `session_flush.sh` | 写入原始对话消息 | 每 ~5 轮，或话题切换时 |
| `session_snapshot.sh` | 写入会话摘要 | session 结束时 |
| `memory_store.sh` | 写入层B长期事实 | 需要保存偏好/决策/项目记录时 |
| 备选：`scripts/memory_backup.sh` | 全量备份 | cron 03:00 |

### 数据库修复（2 个）

| 修复 | 耗时 |
|:----:|:----:|
| FTS5 重建（trigram） | ~10 分钟 |
| `memory index --force` | ~3 分钟 |

### AGENTS.md 更新

更新 Transcript 写入规则，加入 flush + snapshot 双策略指引。
