## 1. 设计哲学

### 1.1 五篇文章的合成结论

| 来源 | 核心观点 | 在本方案中的体现 |
|------|---------|----------------|
| 第一篇：角色冲突 | 一个 Agent 承担多个角色会产生系统性偏差 | 主 Agent 不自审，验证交给独立 Agent |
| 第二篇：上下文中心 | 拆分必须基于上下文是否可隔离，不是模仿人类组织 | 黑盒验证可以拆；连续阶段不拆 |
| 第三篇：验证 Agent 是最稳的起点 | Generator-Verify 是质量闸门，不是主架构 | C+ 模式的验证环节 |
| 第四篇：五种协作模式 | 从最简单开始，出现信号再演进 | 默认 Generator-Verify，必要时用 Orchestrator |
| 第五篇：选型与演进 | 模式是积木不是菜单，可组合使用 | 核心用 Generator-Verify 叠加在基础流程之上 |

### 1.2 三个基本原则

1. **从简单开始** — 不提前搭建复杂架构，当前够用就维持
2. **硬约束兜底，软约束补缝** — 能用工具集隔离的不用 prompt 约束
3. **不信任模型的自觉性** — 能用规则写清楚的绝不留给模型临场判断

## 2. 角色定义与职责

### 2.1 角色总览

```
┌─────────────────────────────────────────────┐
│              先生（决策者）                    │
│  提出需求 → 确认方案 → 做最终决策 → 拍板落地   │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│         主 Agent（我，DeepSeek V4 Flash）      │
│  职责：规划 + 执行 + 产出方案 + 审查指令包     │
│  约束：不自审、不中转审查结果                   │
└────────┬─────────────────┬──────────────────┘
         │                 │
         ▼                 ▼
┌──────────────────┐  ┌──────────────────────┐
│  验证子 Agent     │  │  GLM API（硅基流动）   │
│  (sessions_spawn) │  │  黑盒验证执行者       │
│  常规任务验证      │  │  审查结果直达先生     │
└──────────────────┘  └──────────────────────┘
```

### 2.2 各角色详细职责

| 角色 | 实体 | 职责范围 | 核心约束 |
|------|------|---------|---------|
| **决策者** | 先生 | 提需求 → 确认方案 → 做决策 → 拍板落地方向 | 不做技术判断，做老板决策 |
| **主 Agent** | 我 | 需求分析 → 方案设计 → 任务执行 → 产出审查指令包 | **不自审**、**不中转审查结果**、复杂任务默认走验证 |
| **验证子 Agent** | `sessions_spawn` + GLM API | 黑盒验证 → 输出结构化审查结果 | 仅调用 API 所需权限、固定基线必须逐条过、对抗性验证 |
| **探索子 Agent** | `sessions_spawn` | 只读代码探索 / 信息收集 | 仅只读工具、不能写文件 |

## 3. 协作模式：C+ 模式

### 3.1 标准流程

```
先生 → 主 Agent（我）：任务需求
         │
         ▼
我执行任务 → 产出【方案/产物】+ 【审查指令包】
         │
         ▼
自动触发验证子 Agent（携带审查指令包 + GLM API Key）
         │
         ▼
验证子 Agent 调用 GLM API → 收到审查结果
         │
         ├──→ GLM 原始审查结果（完整原文）
         │         +
         │    我（主 Agent）的回应（如有不同意见）
         │
         ▼
同时发送给先生：
  [GLM 审查原文 — 完整，不可删减改写]
  ─────────────────────────────────
  [我的回应 — 如有不同意见，标注理由]
         │
         ▼
先生决策：
    ├── 通过 → 我落地执行
    ├── 有条件通过 → 先生同意调整后我落地
    └── 不通过 → 打回修改（最多 3 轮）
```

⚠️ **关键约束 — 审查结果直达先生**：
- GLM 原始审查结果必须**完整转发**给先生，不得删减、缩写、改写或选择性转述
- 我（主 Agent）可以在 GLM 原文前面附一段**摘要**，但必须标注"以下为摘要"，且摘要不能替代原文
- 我可以在 GLM 原文后面附加一段**"我的回应"**，标注不同意的条目及理由，但不得修改 GLM 原文

**格式约束（v1.3 新增）**：
- GLM 审查结果必须放在 代码块 中完整粘贴，逐字符复制，不得改写
- 代码块结束后，加一条分隔线 --
- 我（主 Agent）的任何回应文字只能出现在分隔线之后
- 不得在代码块内添加任何注释、解释或修整
- 如果发现自己在"整理"GLM的结果，停下来，直接粘贴原始内容

### 3.2 先生全程的决策点

| 步骤 | 先生做什么 | 不需要做什么 |
|------|-----------|------------|
| 提出需求 | 描述要什么 | 不需要给出技术方案 |
| 接收审查结果 | 看 GLM 原始审查结果（完整原文） | 不需要理解每条意见的技术实现细节 |
| 看我的回应 | 扫一眼我是否对 GLM 有不同意见 | 不需要判断谁对谁错 |
| 决策 | 选"通过 / 有条件 / 打回" | 不需要判断每条审查意见的技术细节 |
| 3 轮未收敛 | 拍一个方向：调标准 / 调方案 / 升级范围 | 不需要纠结具体哪个点 |

**关于审查结果长度的处理**：
- 如果 GLM 原始输出较长，我可以附一份**简洁摘要**方便先生速览
- 摘要必须标注"以下为摘要"，且原文必须同时提供
- 先生可选择先看摘要再决定是否看原文

### 3.3 C+ 模式适用边界

| 条件 | 是否走 C+ | 原因 |
|------|----------|------|
| 任何方案设计类任务 | ✅ 必走 | 方案最容易出现角色冲突 |
| 涉及配置修改的操作 | ✅ 必走 | 安全优先 |
| 涉及外部系统接入 | ✅ 必走 | 风险最高 |
| 涉及金钱的操作 | ✅ 必走 | 财务安全 |
| 代码实现（已有明确方案） | ⚠️ 子 Agent 审即可 | 方案已定，只验证实现是否正确 |
| 简单查询 / 日常提醒 | ❌ 无需审查 | 风险低，不值得走验证流程 |
| 先生明确说"直接做" | ❌ 跳过 | 先生决策权优先 |

## 4. 任务分级与验证路径

### 4.1 三级分类

```
┌──────────┐   ┌──────────┐   ┌──────────┐
│  🔴 红色  │   │  🟡 黄色  │   │  🟢 绿色  │
│  关键任务  │   │  常规任务  │   │  简单任务  │
└──────────┘   └──────────┘   └──────────┘
```

### 4.2 分级标准

| 级别 | 包含范围 | 验证路径 | 先生介入 |
|------|---------|---------|---------|
| **🔴 红色** — 关键任务 | 方案设计 / 架构决策 / 安全配置 / 外部API接入 / 涉及金钱 / 先生指定重要 | 主 Agent → 验证子 Agent（调 GLM）→ GLM 原始结果 + 主 Agent 回应 → 先生决策 | 必须看结果并拍板 |
| **🟡 黄色** — 常规任务 | 代码实现 / 文档输出 / 信息整理 / 图表生成（方案已确定） | 主 Agent → 验证子 Agent（调 GLM）→ GLM 原始结果 + 主 Agent回应 → 先生确认 | 可秒阅即确认 |
| **🟢 绿色** — 简单任务 | 简单查询 / 日常提醒 / 文件读取 / 先生说"直接做" | 主 Agent 直接执行 | 无需介入 |

### 4.3 分级动态调整

- 先生可以随时覆盖默认分级："这个按红色走"或"这个直接做"
- 从实践中积累：哪些类型任务经常需要降级或升级
- 首次出现的任务类型默认按黄色走，后续根据先生反馈调整
- 我产出方案时附带建议分级标签：`[建议分级：🔴/🟡/🟢]`，先生可秒确认或秒改

## 5. 固定审查基线（6条）

> 以下 6 条是每一次验证执行的**最低审查标准**，不可删减。

| 编号 | 检查项 | 老板视角的问法 | 通俗理解 | 审查方法 |
|------|--------|--------------|---------|---------|
| 1 | **需求覆盖** | "我要的事，方案全管到了吗？有没有漏的？" | 别答应了我5件事只做了3件 | 逐条对照原始需求，检查每一条是否都有对应方案 |
| 2 | **内部一致性** | "方案前后说的有没有互相矛盾的地方？" | 前面说用A方案，后面又假设用B | 检查方案的各个部分之间是否存在逻辑冲突 |
| 3 | **可落地性** | "这个方案说要用到的工具、资源，我手上真的有吗？" | 别设计了法拉利的方案，我只有自行车的预算 | 检查方案中涉及的每项资源是否在现有环境中可用 |
| 4 | **异常处理** | "如果出错了怎么办？方案有没有提前想到哪些环节会出问题？" | 别只告诉我顺的时候怎么走，告诉我堵了怎么办 | 检查方案是否覆盖了失败场景 |
| 5 | **过度设计** | "有没有为了看起来全面而加的实际用不上的东西？" | 我要一把螺丝刀，别给我配一整个工具箱 | 检查是否存在当前需求不需要的模块/步骤 |
| 6 | **假设透明** | "方案里有哪些前提假设？如果这些假设不成立会怎样？" | 别默默假设了一个条件没告诉我，结果上线才发现不成立 | 找出方案中的所有隐含假设，评估其风险 |

### 5.1 严重程度与决策权映射

每条问题按以下等级标注，严重程度直接对应默认决策：

| 严重程度 | 含义 | 默认决策 | 谁可以覆盖 | 覆盖条件 |
|---------|------|---------|-----------|---------|
| 🔴 **致命** | 方案核心逻辑错误，无法执行 | 自动"不通过"，必须修改 | 仅先生 | 先生明确说"这个不管，继续" |
| 🟠 **严重** | 方案有明显缺陷 | 自动"有条件通过" | 先生或主 Agent | 主 Agent 给出合理解释，但先生必须看到原始审查意见 |
| 🟡 **一般** | 有优化空间，不影响核心逻辑 | 建议，不阻塞通过 | 主 Agent 可自行判断 | — |
| ⚪ **建议** | 非必须，值得考虑 | 记录备选，不阻塞通过 | — | — |

### 5.2 审查方法论要求

验证子 Agent 的审查方法论必须包含：
1. **对抗性思维** — 主动构造会让方案出错的场景，不是温和地"检查一下"
2. **边界覆盖** — 不能只检查标准流程，必须检查空值、极端参数、异常路径
3. **负向测试** — 假设用户故意输入错误数据或恶意输入，系统能否正确处理
4. **全面检查** — 不能只跑一两个最明显的检查就宣布通过

## 6. 验证子 Agent 技术实现

### 6.1 架构设计

```
主 Agent（我）
    │
    ├── 产出方案 + 审查指令包
    │
    └── sessions_spawn（验证子 Agent）
            │
            ├── 接收：审查指令包（含方案内容）
            ├── 携带：GLM API Key（通过 attachments 注入）
            │
            ├── 调用 GLM API（curl）
            │    POST https://api.siliconflow.cn/v1/chat/completions
            │    Headers: { Authorization: "Bearer <API_KEY>" }
            │    Body: { model, messages, temperature }
            │
            └── 收到审查结果 → 结构化输出
                验证子 Agent 返回：
                ├── 总体结论（通过/有条件通过/不通过）
                ├── 问题清单（带严重程度）
                └── 建议方向
```

### 6.2 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| GLM 模型 | `zai-org/GLM-5.2` 或先生指定的版本 | 硅基流动中转 |
| API 端点 | `https://api.siliconflow.cn/v1/chat/completions` | 硅基流动 API（OpenAI 兼容格式） |
| 验证子 Agent 模型 | **与主 Agent 同等级模型**（不自降级） | 验证需 curl、JSON 解析、格式化，能力不可降级 |
| GLM 审查温度 | `temperature: 0.3` | 降低随机性，提高审查稳定性 |
| 最大审查轮数 | 3 轮 | 超限由先生决策 |
| 审查超时 | 120 秒 | GLM API 响应应在此时间内 |
| 方案过大时分块 | 方案超 4000 tokens 时，按模块拆分，分批次审查 | 确保 GLM 单次审查质量 |

### 6.3 安全性

- API Key 通过 `sessions_spawn` 的 `attachments` 参数注入，不暴露在对话历史中
- 验证子 Agent 没有文件写入权限（默认无 `write`/`edit`/`apply_patch`）
- 验证子 Agent 的任务范围仅限于：读取审查指令包 → 调用 GLM API → 输出结构化结果
- API Key 在一个子 Agent 会话结束后即从上下文释放

### 6.4 调用 GLM API 的方式

验证子 Agent 使用 `exec` 工具执行 curl 命令调用 GLM API（通过硅基流动中转）：

```
curl -s -X POST "https://api.siliconflow.cn/v1/chat/completions" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <API_KEY>" \
  -d '{
    "model": "zai-org/GLM-5.2",
    "messages": [
      {"role": "system", "content": "审查指令包的系统提示词..."},
      {"role": "user", "content": "待审查的方案内容..."}
    ],
    "temperature": 0.3,
    "max_tokens": 4096
  }'
```

## 7. 审查指令包规范

### 7.1 审查指令包的固定结构

每次产出方案时，**主 Agent 同步生成审查指令包**，包含以下三个部分：

#### 第一部分：固定基线（必带，不可删减）

内容为第 5 节定义的 6 条基线。每次照搬，不修改。

#### 第二部分：本次补充的专项验证点

```
建议验证方向：
1. [按任务特性列出的重点验证点]
2. ...
3. ...

已知风险（主 Agent 自查标记）：
- [已知但不确定的点]
- [方案中做了权衡的地方]
- [有条件限制可能出问题的地方]

⚠️ 提示 GLM：以上基于我（主 Agent）自身视角。
GLM 应独立判断是否需要额外验证点，并自主补充我未列出的风险。
```

#### 第三部分：已知设计假设

```
方案中隐含的设计假设：
1. [假设1] → 如果不成立，影响是...
2. [假设2] → 如果不成立，影响是...
...
```

### 7.2 审查输出格式（GLM 强制要求）

```
总体结论：通过 / 有条件通过 / 不通过

一、固定基线审查结果
  1. 需求覆盖：[通过/不通过] + 严重程度 + 理由
  2. 内部一致性：[通过/不通过] + 严重程度 + 理由
  3. 可落地性：[通过/不通过] + 严重程度 + 理由
  4. 异常处理：[通过/不通过] + 严重程度 + 理由
  5. 过度设计：[通过/不通过] + 严重程度 + 理由
  6. 假设透明：[通过/不通过] + 严重程度 + 理由

二、专项验证点审查结果
  [按主 Agent 建议的验证方向逐条检查结果]

三、GLM 独立补充发现
  [主 Agent 未列出的高风险点]
  [防止偷懒的对抗性验证结果]

四、问题清单
  | 问题描述 | 严重程度 | 对应基线/专项 | 修改建议 |

五、汇总统计
  - 通过项：X 条
  - 不通过项：X 条
  - 独立补充发现：X 条
```

## 8. 迭代规则与人工介入

### 8.1 标准迭代流程

```
第 1 轮：我产出方案 → 验证子 Agent 审查 → 输出结果
  ├── 通过 → 先生确认 → 落地
  ├── 有条件通过 → 我按意见调整 → 先生确认 → 落地
  └── 不通过 → 进入第 2 轮

第 2 轮：我按审查意见修改 → 验证子 Agent 复查
  ├── 通过 → 先生确认 → 落地
  ├── 同一条问题再次出现 → 触发先生介入判断
  ├── 我不同意某条意见 → 在回应中标注理由，先生裁决
  └── 不通过（新的问题） → 进入第 3 轮

第 3 轮：我再次修改 → 验证子 Agent 复查
  ├── 通过 → 先生确认 → 落地
  └── 仍不通过 → 暂停 → 先生介入
```

### 8.2 人工介入触发条件

| 条件 | 操作 |
|------|------|
| 达到最大 3 轮审查未通过 | 暂停，先生介入 |
| 我和 GLM 对同一问题连续 2 轮有分歧 | 先生介入，拍一个方向 |
| 我和 GLM 在第 **1 轮**对某条意见有分歧 | 在回应中标注"我不同意，理由是XXX"，先生看到双方意见后决定 |
| 先生觉得"这个方向不对，换思路" | 先生直接叫停，重新定义需求 |
| 先生觉得"差不多可以了" | 先生有权跳过审查，直接通过 |

> **关键规则**：我（主 Agent）如果不同意 GLM 的某条审查意见，**不能自行忽略**。必须在"我的回应"中明确标注不同意及理由，由先生看到双方意见后决定。

### 8.3 先生介入后的决策选项

| 选项 | 适用场景 |
|------|---------|
| **调标准** | GLM 过于苛刻，可放宽某些基线要求 |
| **调方案** | 确实有问题，需要重新设计 |
| **升级范围** | 当前方案已不够，需要重新讨论需求 |
| **直接通过** | 先生认为风险可控或时间紧迫 |

## 9. 子 Agent 使用原则

### 9.1 什么时候用子 Agent

| ✅ 适合用 | ❌ 不适合用 |
|----------|-----------|
| 独立探索（查文档、查代码） | 同一工作的连续阶段（规划→实现→测试不要分拆） |
| 并行调查多个互不依赖的问题 | 需要频繁对齐理解的任务 |
| 有清晰接口的黑盒任务 | 接口不明确的任务 |
| 会大量占用上下文窗口的探索性工作 | 和主任务共享大部分上下文的任务 |

### 9.2 子 Agent 的标准交互协议

```
子 Agent 任务包：
├── 任务名称：（简短明确）
├── 任务描述：（清晰的边界声明）
├── 上下文材料：（文件列表 / 代码片段 / 已知约束，不超过必要量）
├── 工具权限声明：（能读什么、能写什么、能运行什么）
└── 输出格式要求：（结构化的结果模板）
```

### 9.3 子 Agent 的权限分级

| 子 Agent 类型 | 允许的工具 | 禁止的工具 |
|--------------|-----------|-----------|
| 验证子 Agent | exec（仅 curl 调 GLM API）、web_fetch | write、edit、apply_patch、cron、message |
| 探索子 Agent | read、exec（仅只读命令：grep、find、cat、git log） | write、edit、apply_patch 及任何删除操作 |
| 执行子 Agent | 按任务需要开放，先生确认 | 默认仅开放必要的 |

### 9.4 子 Agent 的生命周期管理

| 管理项 | 规则 |
|--------|------|
| 创建 | 仅由主 Agent 通过 `sessions_spawn` 创建 |
| 存活期 | 单次任务：任务完成即结束（`cleanup: "delete"`） |
| 最大并行数 | 同时不超过 3 个（防止 token 浪费） |
| 完成通知 | 自动在完成时汇报到主 Agent |
| 异常 | 超时或失败 → 主 Agent 决定重试或上报先生 |

## 10. 主 Agent 默认行为规则

### 10.1 不自审 + 不中转审查结果规则

```
❌ 禁止：我产出方案后自己执行审查
❌ 禁止：我缩写、删减、选择性转述 GLM 的审查结果
✅ 正确：复杂任务走验证子 Agent，审查结果直达先生
```

### 10.2 第一版标注规则

所有方案类交付物，第一版必须标注：

> **初稿，待审查**
>
> 建议分级：[🔴/🟡/🟢]
> 审查状态：[已完成 / 审查中 / 待审查]
> 审查结果：[通过 / 有条件通过 / 不通过]
> 已知风险：[主要风险点摘要]

### 10.3 复杂度判断规则

| 信号 | 行为 |
|------|------|
| 需要产出跨模块的方案 | 自动启动验证流程 |
| 涉及配置修改或权限变更 | 自动标注为红色任务 |
| 先生说的"这个你看看怎么设计" | 默认走完整流程 |
| 上下文窗口使用率 > 60% | 考虑用子 Agent 做探索性工作 |
| 需要查大量信息再决策 | 先 spawn 探索子 Agent 收信息 |

### 10.4 成本意识规则

| 规则 | 说明 |
|------|------|
| 验证子 Agent 用同等级模型 | 验证环节需要准确性，不自降模型等级 |
| GLM API 调用控制温度 | 审查场景用 `temperature: 0.3`，减少随机性 |
| 方案过长时按模块分块审查 | 确保 GLM 单次审查质量，分块后分别调用 |
| 最大审查轮数 3 轮 | 超过即上报，不无限制循环 |
| 探索子 Agent 可用更便宜的模型 | 探索阶段不需要强推理能力，速度和成本更重要 |

## 11. 异常处理与安全

### 11.1 异常处理

| 异常情况 | 处理方式 |
|---------|---------|
| **GLM API 不可用**（网络超时） | 重试 2 次，间隔 5 秒；仍失败 → 报告先生"GLM 暂时不可用，是否用子 Agent 替代审查" |
| **API 返回错误**（认证失败、额度不足） | 立即上报先生，同时检查 API Key 是否过期 |
| **API 返回内容不完整/截断** | 重试并增加 max_tokens；仍截断 → 报告先生"审查不完整，建议手动重试" |
| **GLM 输出格式不符合规范** | 按结构化模板手动重新整理；缺失信息标注为"GLM 未提供"；无法整理时记录为"审查不完整" |
| **验证子 Agent spawn 失败** | 重试 1 次；仍失败 → **降级为手动模式**：将审查指令包 + 方案发给先生，先生自行转发给 GLM |
| **先生不在电脑前需手动模式时** | 我给出审查指令包，先生复制到 Cherry Studio 发给 GLM，GLM 输出再发回来 |

### 11.2 安全规则

1. **API Key 安全**
   - GLM API Key 不写入任何文件，不发送到对话历史
   - 仅通过 `sessions_spawn` 的 `attachments` 参数注入
   - 子 Agent 执行完毕后 Key 自动释放

2. **工具权限安全**
   - 验证子 Agent 仅有调用 API 所需的最小权限
   - 探索子 Agent 仅有只读权限
   - 主 Agent 保持全工具但遵守不自审和中转禁令

3. **隐私安全**
   - 验证子 Agent 不存储任何先生信息
   - GLM API 调用内容仅限于待审查的方案本身
   - 审查结果仅在会话期间使用，不持久化到第三方

4. **API Key 归属**
   - 目前通过硅基流动中转调用 GLM-5.2，API Key 归属先生个人账户
   - 费用走先生硅基流动账户余额

### 11.3 执行纪律

| 规则 | 值 |
|------|-----|
| 单次任务最长执行时间 | 30 分钟，超时暂停并报告 |
| 同一任务最大失败次数 | 3 次，超限停止并报告 |
| 估算费用超过 ¥30 | 暂停并询问 |
| 最大审查轮数 | 3 轮，超限先生介入 |

## 12. 已知假设

> 本方案基于以下假设设计。假设若部分不成立，需调整对应环节。

| 隐含假设 | 风险 | 应对方案 |
|---------|------|---------|
| GLM API 会按格式输出结构化结果 | LLM 不保证格式 100% 合规 | 格式异常时按 11.1 节降级处理（手动整理） |
| 主 Agent 能不选择性转述 GLM 结果 | 模型天然倾向维护自身方案 | 已改流程为 GLM 结果直达先生 + v1.3 代码块格式约束（半硬约束，降低中转风险） |
| 先生有精力阅读 GLM 原始输出 | 非程序员可能觉得长报告太累 | 我提供"摘要"先行，先生选择是否看原文，但不能只要摘要不要原文 |
| 单次 GLM API 调用足以完成完整审查 | 复杂方案可能需要多轮 | 方案 > 4000 tokens 时按模块拆分审查 |
| sessions_spawn 能成功创建验证子 Agent | 平台可能存在限制 | spawn 失败时降级为手动模式 |
| 腾讯云服务器可访问硅基流动 API | 存在网络限制或防火墙 | 若 API 不可达，通知先生并改用手动模式 |
