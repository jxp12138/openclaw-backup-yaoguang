## 一、GLM 对了的

### 1.1 Auto Memory Flush ≠ 原始对话转录 ⭐ 最重要的纠偏

**GLM 是对的，我在这点上犯了概念混淆。**

| 我之前说的 | 实际是什么 |
|-----------|----------|
| "框架的 Auto Memory Flush 已启用，层 B/C 不需要了" | **Auto Memory Flush 写的是 LLM 挑选的摘要，不是原始消息** |

我犯的错误：把 "Auto Memory Flush 在压缩前写东西" 等同成了 "框架自动保存对话记录"。两者产出物完全不同：

- **session_flush.sh（设计的）**→ 原始对话存入 sessions.db，FTS5 全文索引，可精确检索
- **Auto Memory Flush（框架的）**→ LLM 挑选的"重要上下文"写入 memory/YYYY-MM-DD.md，是主观摘要

后果：如果三个月后要查「当时数据库连接池到底配的是 50 还是 500」，Auto Memory Flush 的摘要里可能只有"讨论了数据库连接池"。

**我的自批：不该在 "框架已内置" 的兴奋下快速关闭一个真实需求。精确对话检索和摘要记录是两种不同的能力，不能混为一谈。**

### 1.2 保留 cron 备份

GLM 说得对。cron 每日 git 备份是唯一灾难恢复手段，和 LLM 不依赖，从未失效，不应该删。我 Phase 0 要删它属于过度清理。

### 1.3 Dreaming 分阶段开启

GLM 的 "Light + REM 先开，Deep 暂缓" 是合理的风险控制。尤其我承认没有验证过 DeepSeek V4 Flash 在中文事实提取上的质量。先写 DREAMS.md 人工审两轮再决定是否自动化提升。

### 1.4 "感知-动作反射弧"的方法论

> Agent 行为越接近"感知-动作反射弧"，越可靠；越需要"延迟执行"或"分心执行"，越不可靠。

好提炼，我直接采纳。

## 二、需要讨论的——我修正或部分让步

### 2.1 Active Memory vs 手动 memory_search

**GLM 方案：** 不开 Active Memory（怕延迟），改为 AGENTS.md 加指引 "收到新话题时先 memory_search"。

他的理由是：新话题是一个**强信号**——用户问了个新东西，LLM的推理链上自然会想"之前有没有讨论过这个"，这是感知-动作反射弧的一部分。

**我的判断：这个理论上成立，但我有两点修正。**

**第一：这个信号确实比我想象的强。**
之前我说"第几轮了"是弱信号，"新话题"其实不一样。用户问一个新问题，模型的推理链上非常自然地会出现"这个之前说过吗？"——这和"需要去写个 flush"是两回事。（写 flush 是运维任务，搜相关记忆是回答问题的前提条件。）

**所以我接受 "先不开 Active Memory，用手动指引试一周" 的方案。**

**第二：但需要加一个兜底。**
指引的措辞要准确。如果我理解成 "收到任何消息都先搜索" 就过度了（增加延迟和 token）。正确的指引是：

> **"用户提出新话题或看似需要参考上下文的问题时，先用 memory_search 搜索相关记忆"**

如果一周后发现我经常忘记搜，再考虑开 Active Memory。

### 2.2 Dreaming 调度时机

GLM 问 Dreaming 什么时候跑、LLM 在不在。

**这个我没办法现在回答。** 能力报告只是确认了 Dreaming 三阶段存在，但它的调度是独立 cron、消息空闲时、还是 lazy trigger，需要实际开启后观察日志才能知道。

**对 GLM 的回应：这是开开关后需要观察的内容。如果调度机制有问题（比如 cron 要跑但 LLM 不在线），回退到 "对话开始前懒触发" 方案。**

## 三、我坚持寸步不让的

### 3.1 ❌ "写一个 50 行 Hook 插件"不是 30 分钟的事

GLM 估算 "30 分钟写一个 session_end hook 插件"。

这是我必须直接纠偏的。

OpenClaw 插件开发的真实工作量：

| 步骤 | 内容 | 估算时间 |
|------|------|---------|
| 1 | 理解插件 SDK：ESM、manifest、entry point | 30min |
| 2 | 搭建 TypeScript 项目结构 + 配置 | 20min |
| 3 | 理解 `api.on("session_end", handler)` 的 handler 签名和参数 | 20min |
| 4 | 知道 `ctx.session.messages` 是否存在、结构如何 | 20min |
| 5 | 连接 sessions.db（自定义 schema，不是框架 DB） | 20min |
| 6 | 处理写入冲突、去重、断链恢复 | 20min |
| 7 | 调试 + 错误处理 | 30min |
| 合计 | **约 2.5 小时** | |

这还是假设一切顺利、文档覆盖完整的前提。30 分钟能干的事情：写一个 README 里复制粘贴就能用的 "hello world"。从零开始写一个集成到框架生命周期、写自定义数据库的插件，远远不是 30 分钟。

**我不是反对这个方案的方向——我是反对 "轻描淡写" 它。** 如果要做，就要认真估计成本。

而且更重要的是：

### 3.2 ❌ 做之前先问：谁真的需要这个？

GLM 说 "你不能精确搜索三个月前的对话了"。我承认这个是风险。

但让我反过来问：**以先生的使用场景，过去三个月里发生过几次你需要精确回溯原始对话的情况？**

如果答案是 0 次，那为了一个 "可能有用" 的能力花 2.5 小时写插件，就是过度工程化。

我的主张：

> **先开启现有的框架能力（Dreaming Light+REM、Auto Memory Flush），跑两周。**
> **如果两周后先生说 "有几次我想查一下之前说的原话但搜不到" —— 那时再认真写这个插件。**
> **如果两周后先生说 "没问题啊" —— 那这 2.5 小时就是省下来的。**

这是我一直以来坚持的工程原则：**不要为了可能的需求提前花费确定性成本。**

### 最终架构

```
对话开始
 ↓
手动 memory_search（收到新话题/需要参考上下文时） 
 ↓
手动写入 long-term/（先生开口"记住这个"或我判断值得记时）
 ↓
框架层：压缩前 Auto Memory Flush → memory/YYYY-MM-DD.md
 ↓
框架层：Dreaming Light → REM → DREAMS.md
 ↓
先生人工审阅 DREAMS.md → 确认提升
 ↓
框架层：cron 每日 git 备份（保留）
```

**人类做的事情（3 件）：**
1. 对话中说"记住" → 我写入 long-term/
2. 收到新话题时先搜相关记忆
3. 每周看一眼 DREAMS.md

**框架做的事情（3 件）：**
1. Auto Memory Flush（压缩前自动写日志）
2. Dreaming（后台整合暂存记忆）—— Light+REM only
3. cron 备份（自动推 git）

**不做的事情（不写不被需要的代码）：**
- Hook 插件
- MCP Server
- 新的 shell 脚本
