## 一、事实核查：能力边界调查的关键发现

子代理全面扫描后，发现了几个**颠覆之前判断**的事实：

### 最重要的发现：框架内置能力远超预期

| 我们以为 | 实际上 |
|---------|--------|
| 需要自己写 layer B/C 脚本 | ✅ **memory-core 已内置完成** |
| 需要设计触发机制 | ✅ **压缩前 Auto Memory Flush 已启用** |
| 需要实现记忆整合 | ✅ **Dreaming 系统（三阶段）已内置，只差 enable** |
| 需要自己写缓存 | ✅ **session-memory hook 已内置** |
| 需要为记忆召回写子代理 | ✅ **Active Memory 插件已存在，只差 enable** |

### 当前已启用但未利用的框架能力

```
memory-core (已启用)
├── memory_search / memory_get ✅
├── Memory Flush（压缩前） ✅ 自动
├── Dreaming（三阶段整合） ❌ 未启用 ← 开个开关就行
├── Active Memory（回复前召回） ❌ 未启用 ← 开个开关就行
└── Memory Wiki ❌ 未启用
```

**核心结论：我们花精力设计的东西，框架已经帮你做好了，就差激活。**

## 二、对 GLM 建议的逐一回应

### GLM 说「每一轮自动 flush DB」
**我的判断：基本同意，但不必自己写**
- 框架的 Auto Memory Flush 在压缩前触发，不是每一轮
- 如果要"每一轮"，可以用插件 `message_sending` hook 拦截写入
- 但压缩前 flush 对个人使用场景来说频率已经足够
- ⭐ 关键发现：OpenClaw 有**完整的插件 SDK**，可以写自定义 hook 插件

### GLM 说「memory_store → MCP Tool」
**我的判断：技术上可行，但暂时没必要**
- OpenClaw 支持 MCP 客户端（stdio / SSE / streamable-http）
- 可以注册外部 MCP Server 提供 `memory_store` 工具
- 但当前 `edit`/`write` 工具直接写 `long-term/` 的摩擦已经极低
- **MCP 的更大价值在于接入外部知识库或数据源**，而不是替代文件写入

### GLM 说「Layer A 之所以工作是因为执行模型碰巧对了」
**我的判断：80% 对，但少说了一个关键**
- 对的部分：执行模型确实是根因（LLM 无法可靠地执行模糊指令）
- 漏说的部分：**触发信号的清晰度**才是决定性因素
  - 「记住这个」= 强信号 → 100% 执行
  - 「大概第几轮了」= 弱信号 → 从未执行
  - 框架层（memory-core）提供的就是清晰信号——压缩是确定事件，不是模糊指令

### GLM 提出的方案 C（框架层自动化）
**我的判断：正确路线，而且已经走了一半**
- memory-core + dreaming + Active Memory 正好构成他说的「框架层自动」
- 方案 C 的重点不是写代码，是**开开关**

## 三、修正后的解决方案

### 不再重构，直接点亮

```
第〇步：停止写自定义脚本（不做 layer B/C 了）
第一步：开启 Dreaming
第二步：开启 Active Memory
第三步：保持 Auto Memory Flush（已启用）
第四步：清理 dead scripts
```

### 具体操作

#### Phase 0 — 清理（5 分钟）

1. 删除 `scripts/session_flush.sh`、`session_snapshot.sh`、`memory_store.sh`、`session_auto_flush.sh`
2. 删除 `scripts/daily-git-backup.sh`
3. 删除 `memory/pending-memory.md`、`long-term/pending/` 目录
4. 精简 AGENTS.md 中记忆系统的章节（删掉旧的 layer B/C/D 描述）
5. 移除无用的 cron jobs（daily-git-backup、auto-dream）

#### Phase 1 — 点亮框架能力（10 分钟）

1. **开启 Dreaming**
   - `openclaw config set plugins.entries.memory-core.config.dreaming.enabled true`
   - 可选：设置 `dreaming.schedule` 调度频率

2. **开启 Active Memory**
   - `openclaw config set plugins.entries.active-memory.enabled true`
   - 或 `openclaw plugins enable active-memory`

3. **确认 Auto Memory Flush 已启用**
   - 检查 `agents.defaults.compaction.memoryFlush.enabled`（默认为 true）

4. **配置 Memory Wiki（可选）**
   - 如果先生使用 Obsidian，可以开启

#### Phase 2 — 观察与微调（持续）

1. 观察几轮对话后 `memory/YYYY-MM-DD.md` 是否自动写入
2. 查看记忆搜索效果是否有改进
3. 根据实际效果微调 dreaming 配置（频率、模型、深度阈值）

## 四、架构图（新）

```
对话开始
  ↓
用户输入 → Active Memory（回复前检索相关记忆） ← 注入上下文
  ↓
回复中 → memory_search / memory_get（按需检索）
  ↓
对话中 → 我手动写入 long-term/（Layer A）
  ↓
压缩前 → Auto Memory Flush → memory/YYYY-MM-DD.md
  ↓
（后台）Dreaming Light → REM → Deep → MEMORY.md 提升
  ↓
对话结束 → （什么都不用做，框架自动处理了）
```

**三行搞定，不再需要任何自定义脚本。**
