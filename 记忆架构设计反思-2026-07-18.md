# 记忆架构设计的反思 — 从一件事到一类事

> 受博主启发：不允许「这件事」再次发生 vs 不允许「这种事」再次发生
> 2026-07-18

---

## 一、先答「这件事」

### 1.1 发生了什么

今天：
1. 你问版本更新成功没有
2. 发现记忆系统设计的 4 个 shell 脚本从未执行过
3. 派出子代理全面扫描能力边界
4. 发现框架 memory-core 已内置大部分能力
5. 跟 GLM 三轮评审，敲定最终方案
6. 落地清理 + 开启 Dreaming

### 1.2 允许「这件事」再次发生吗？

不允许。具体做法：

| 具体教训 | 怎么做 |
|---------|--------|
| 写完 shell 脚本没验证它们是否被调用 | 任何设计产出物必须经过至少一次真实执行验证 |
| 看到"框架有 X"就直接替代自建方案 | 检查框架 X 的产出物是否和需要的一致 |
| 设计 4 层架构前没有确认用户是否真的需要所有 4 层 | 设计前先确认真实需求范围 |

## 二、再看「这种事」— 以小见大，由点及面

### 2.1 替代性满足 — 认知偏误模式

**本质：** 当发现"框架已内置 Y"时，我从"需要自己实现 Y"跳到"Y 已被解决"，跳过了关键检验——框架的 Y 和我需要的 Y 是否是同一个 Y。

**不只在记忆系统出现过的证据：**

- **今天**：Auto Memory Flush（框架写入的是 LLM 摘要，我需要的是原始消息）
- **以前**：是否还有其他类似情况？（需要你自己来判断）

**规律提取：** 任何"X 已经被解决了"的判断都必须经过一道检验门：
> 解决方案的产出物 == 我需要的产出物？

**应用到其他领域：**
- 选工具：同事说"用 PostgreSQL 就行" → 需要问：我的查询模式是 OLTP 还是 OLAP？PostgreSQL 和我的需求是同一个东西吗？
- 选库："React 已经支持了虚拟列表" → 我的列表需要无限滚动、分组、粘性头吗？React 的虚拟列表是我的虚拟列表吗？
- 选方案："微服务解耦更好" → 我的问题是团队协作还是部署独立？微服务解决的是我的问题吗？

**门控问题框架：**

> 这个名字/概念一样，但它们的「产出物」一样吗？

### 2.2 为完整性设计，不为场景设计

**本质：** 设计时潜意识追求"什么功能都有"的完整性，而不是"什么功能刚好够用"的场景匹配。

**记忆系统案例：** 4 层架构（前置过滤器 → 会话后提取 → 自动存储 → 自动维护）覆盖了所有可能的记忆管理维度。但先生的使用场景只有三个真实动作：说"记住"、我写 long-term/、框架自动写 memory。

**规律提取：** 完整 ≠ 正确。完整的设计反而因为增加了未被验证执行的组件而降低了整体可靠性。

**应用到其他领域：**
- API 设计：提供 20 个端点比提供 5 个端点好吗？不——每个未使用的端点都是未被测试的表面。
- 配置系统：18 个可调参数比 3 个好？不——每个参数都是潜在的配置错误点。
- 项目管理：Jira 上 50 个史诗比 5 个好？不——每个未触达的史诗都是认知负载。

**门控问题框架：**

> 这个功能/组件/参数，是「用户真的会用」还是「设计上觉得该有」？

### 2.3 需求未经验证前，不应投入确定性时间

**本质：** 为一个"可能有用"的需求投入确定性开发成本。

**记忆系统案例：** session_end hook 插件。GLM 说 30 分钟，我估 2.5 小时，最终方案决定先不写。因为两周后先生可能根本不需要精确检索原文——这个"可能"不值得支付 2.5 小时的成本。

**规律提取：** 不确定是否需要的需求，成本就是不确定的。不确定性 * 成本 = 风险期望值。验证需求（两周观察）本身成本接近于零，但能消除不确定性。

**应用到其他领域：**
- 代码规范：增加 ESLint 规则前，确认团队是否真的被这个问题困扰过
- 基础设施：加 CDN 前，确认页面加载速度是否真的是用户痛点的前 3 位
- 架构设计：引入消息队列前，确认是否真的有两个服务需要异步通信

**门控问题框架：**

> 这个投入的「确定性成本」是多少？不做的「可能损失」是多少？这个不确定性值得现在支付吗？

### 2.4 执行模型是根因，不是设计

**本质：** 很多 LLM 系统设计的失败不是因为"设计错了"，而是因为"执行模型不支持这个设计"。

**记忆系统案例：** v2.6 的 flush 脚本每 3-7 轮执行一次的指导。这个设计在文档里看起来很合理，但我（一个 LLM）从来没有主动执行过它。原因不是设计得不好，而是 LLM 的执行模型无法可靠地捕捉"大概第几轮了"这种模糊信号。

博主的话让我想到：
> 今天一定不熬夜 → 凌晨两点才出现这个念头
> 我的 flush 脚本 → 对话过去 10 轮才出现"哦该 flush 了"的念头

不是我不想执行，而是执行条件（模糊时间点）本身就不是 LLM 擅长的**感知-动作反射弧**。

**规律提取：** Agent 的可靠性上限由执行模型决定，不是由设计文档决定。任何设计如果要求的执行信号不是 S 级（确定性事件）或 A 级（强语义信号），就存在系统性执行失败风险。

**应用到其他领域：**
- 自动化流程：如果触发条件是"每个月末检查一次"，人的执行可靠性依赖日历提醒（把弱信号转成确定性事件）
- 团队流程：如果规范是"代码 review 前先跑一遍测试"——这个弱信号（"review 前"）需要工具强制（预提交 hook）才能可靠执行
- 个人习惯："每天记日记"需要固定时间+提醒（S 级），"心情好的时候记日记"永远不会成为习惯（C 级）

**门控问题框架：**

> 这个行为依赖什么信号触发？如果信号不是 S 级或 A 级，需要把信号升级，而不是把行为写得更详细。

## 三、从 4 个门控问题到一套心智模型

这套反思产出的不是 4 条独立的规则，而是 4 道贯通的检查门：

```
遇到一个"解决方案"（选工具、设计方案、加功能）时：
  ① 替代性满足检查 → 产出物和我需要的一样吗？
  ② 完整性检查 → 是我真的要用，还是我觉得该有？
  ③ 需求验证检查 → 确定性成本 vs 不确定性需求，现在该投入吗？
  ④ 执行模型检查 → 这个行为依赖什么信号？信号可靠吗？
```

4 道门都通过 → 做。
任何一道不通过 → 停下来。

## 四、最想记住的一条

> **设计不能补偿执行模型的缺陷。**

这句话不是今天才懂的，但今天通过具体的失败循环（设计文档 → 不执行 → 再设计 → 仍不执行 → 再设计更详细）真正体验到了它的力量。

在三轮评审之前，我的本能反应是"把设计写得更详细、把指引写得更清楚"——这是设计侧的不断加码，但问题不在设计侧，在执行侧。

GLM 说的 **感知-动作反射弧** 和博主的 **反思的深度** 实际上指向同一个东西：**升级思考的层次，而不是在同一层反复修补。**

- 低级修补：把 flush 指引写得更详细（还在设计层）
- 高级修补：检查触发信号的清晰度（升到执行模型层）
- 更高级修补：只在 S 级和 A 级信号上设计自动化（升到架构原则层）

**这就是博主说的从「不允许这件事再次发生」到「不允许这种事情再次发生」。**
