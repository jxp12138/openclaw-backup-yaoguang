## 一、我们要做什么

在主 Agent（我，腾讯云轻量服务器 Ubuntu）上设置维护子 Agent，通过 cron 定时任务 + sessions_spawn 自动执行服务器日常维护。

**你全程通过微信收报告和做决策。**

---

## 二、整体架构

```
你（微信）→ 主 Agent（我）
              │
              ├── 日常对话（不受影响）
              │
              └── cron 定时触发
                      │
                      ├── [每天 08:00] 健康检查子 Agent
                      │   → 磁盘/内存/CPU/进程/SLL/网络
                      │   → 报告每日状况（健康/不健康都报）
                      │
                      ├── [每周一 08:00] 日志清理子 Agent
                      │   → logrotate 轮转日志
                      │   → 清理 /tmp 过期文件
                      │
                      ├── [每周六 08:00] 安全更新检查子 Agent
                      │   → apt 检查安全更新
                      │   → 列出待安装的更新清单
                      │
                      ├── [每周日 20:00] 配置备份子 Agent
                      │   → 备份关键配置文件
                      │
                      └── [每周日 20:00] 周报汇总
                          → 汇总本周服务器状况变化
```

## 三、定时任务表

| 任务名 | 执行内容 | 频率 | 时间 | 需 sudo |
|--------|---------|------|------|---------|
| health-check | df/free/ps/journalctl/检查nginx&openclaw进程/SSL证书/网络连通性 | 每日 | 08:00 | ✅ journalctl |
| log-cleanup | logrotate 轮转 + 清理 /tmp 过期文件 | 每周一 | 08:00 | ✅ logrotate |
| security-check | apt list --upgradable | 每周六 | 08:00 | ❌ 只读 |
| config-backup | 备份 ~/.openclaw/ 关键配置 + agent-collaboration-rules.md | 每周日 | 20:00 | ❌ 用户目录 |
| weekly-summary | 汇总本周健康、异常、变更 | 每周日 | 20:00 | ❌ |

## 四、消息报告机制

### 每日报告（08:00 健康检查后）

格式：
```
📋 7月11日 服务器日报
✅ 健康：磁盘62% | 内存48% | CPU占用12% | OpenClaw正常运行
检查项：进程存活✅ SSL证书✅ 网络连通✅
``````
或

```
⚠️ 7月11日 服务器日报 — 有异常
❌ 磁盘使用率 87%（超过85%阈值）
✅ 其他项正常
```

### 紧急告警（实时）

| 条件 | 行为 |
|------|------|
| 磁盘 > 85% | 立即通知先生 |
| 关键进程（OpenClaw）宕了 | 立即通知先生 |
| API Key 失效 | 立即通知先生 |

### 每周汇总（周日 20:00）

格式：
```
📊 本周服务器状况变化
正常天数：5天
异常天数：2天（7月8日 磁盘87%，7月10日 网络波动30秒）
已处理：磁盘清理后降至72%
待处理：无
```

## 五、异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| cron 任务未触发 | OpenClaw 框架保证任务调度，极少发生。如连续3次无报告 → 先生介入 |
| 子 Agent spawn 失败 | 重试 1 次；仍失败 → 先生"维护子 Agent 今日未运行" |
| 维护命令执行出错 | 捕获错误输出，连同"出错了"一起发给先生 |
| 磁盘已满 cron 任务跑不了 | 子 Agent 启动就检测 → 立即上报先生"磁盘满，无法执行任务，请手动清理" |
| 先生不看消息 | 消息已发到微信，先生有空时看。紧急告警无法保证即时回应 |

## 七、权限隔离

| 命令类别 | 允许 | 禁止 |
|---------|------|------|
| 系统状态 | df/free/ps/top/uptime/journalctl --no-pager | — |
| 网络 | curl/ping/ss/openssl s_client | — |
| 文件操作 | find(只读)、du、ls | rm/mkfs/shutdown/find -delete |
| 日志管理 | logrotate（配置好的） | 不手动 rm 日志 |
| 备份 | tar/cp/mv（限定路径） | 不允许写系统文件 |
| 包管理 | apt list（只读） | apt upgrade 不自动执行 |

## 八、演进路径（修订版）

修复了 GLM 上轮审查指出的矛盾：

```
第一阶段（当前）→ 第二阶段（条件出现时）→ 第三阶段（未来）
cron + 子Agent    独立 Agent（维护任务        Agent Teams
                  变得复杂、需要独立           （多个 Agent 长期协作
                  workspace 和 auth 时）       维护一个复杂系统）
```

第一阶段和第二阶段不存在非此即彼的逻辑矛盾。第一阶段是**默认方案**，第二阶段的触发条件是：**维护任务种类超过 6 项、或需要长期独立监控、或需要独立身份认证**时才升级。
