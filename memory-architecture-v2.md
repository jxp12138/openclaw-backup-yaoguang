# 🧠 记忆系统架构 v2（个人定制版）

> 融合：原四层系统 + Cloud Code 分类哲学 + 实际运行环境（个人 agent / WSL2 / DeepSeek V4 Flash）
> 最后更新：2026-07-07

---

## 〇、核心原则

| # | 原则 | 含义 |
|---|------|------|
| 1 | **轻量可执行** | 每个规则都是明天能直接用的，不搞过度设计 |
| 2 | **记忆是代码的补集** | 工具/代码能实时查的，不进记忆 |
| 3 | **洞察 > 日志** | 活动日志不是记忆，从活动中提炼出的洞察才是 |
| 4 | **先否决后判断** | 先过"不该记"清单，再考虑"该不该记" |
| 5 | **宁可少记不可错记** | 不确定的一律进暂存池，低价值条目不如不记 |

## 一、全景架构

```
┌──────────────────────────────────────────────────────────────────┐
│                    第零层：前置过滤器（对话中实时）                  │
│  Step 1 - 否决清单 → 命中5类跳过                                   │
│  Step 2 - 洞察检查 → 活动日志跳过，洞察继续                         │
│  Step 3 - 触发条件 T1-T7 → 未命中等会话后补漏                       │
│  Step 4 - 类型识别 + scope → 确定写向哪个文件                      │
│  Step 5 - 冲突检测 → 查重/更新/过时标记/覆盖                       │
├──────────────────────────────────────────────────────────────────┤
│                    第一层：会话后提取（对话结束时）                   │
│  全量扫描本次对话 → 补漏前置过滤器未命中但有价值的条目               │
│  权限：只读对话 + 写记忆目录，不执行副作用操作                       │
│  游标：记录上次处理到哪条消息，下次只看新增内容                      │
│  互斥：主模型已写的记忆不再重复提取                                 │
├──────────────────────────────────────────────────────────────────┤
│                    第二层：自动存储（按类分级写入）                   │
│  按记忆类型 + 时效 + scope → 写入对应文件                          │
├──────────────────────────────────────────────────────────────────┤
│                    第三层：上下文加载（每次任务前 / 每轮对话）        │
│  Phase 1 - 扫描：递归读记忆目录，只读前 30 行元数据（≤200 个文件）  │
│  Phase 2 - 摘要：拼成清单（类型+标签+时间+一句话描述）              │
│  Phase 3 - 选择：当前输入 + 清单 → 选 ≤5 条最相关                   │
│  Phase 4 - 后校验：确定性检查文件名是否真实存在                      │
│  Phase 5 - 注入：读取选中文件全文（≤4KB/条），附时间标签             │
├──────────────────────────────────────────────────────────────────┤
│                    第四层：定期维护（每周复盘）                       │
│  暂存池清理（超7天丢弃）                                           │
│  MEMORY.md 从 daily notes 同步更新                                │
│  过时条目标记清理                                                  │
│  检查是否需重建 embedding 索引                                     │
└──────────────────────────────────────────────────────────────────┘
```

## 二、4 类记忆类型

| 类型 | 定义 | 存放位置 | scope | 标签格式 |
|------|------|---------|-------|---------|
| **🧑 用户记忆** | 先生是谁、偏好/习惯、技术背景 | `MEMORY.md` | `[个人]` | `[user]` |
| **💬 反馈记忆** | 纠正/肯定记录 | `~/self-improving/corrections.md` | `[个人]`或`[项目]` | `[feedback]` |
| **📋 项目记忆** | 项目背景/动机/时间线/截止日期 | `~/self-improving/projects/<name>.md` | `[项目]` | `[project]` |
| **🔗 引用记忆** | 外部资源索引（去哪找信息） | 就近存放 | `[个人]`或`[项目]` | `[reference]` |

### 反馈记忆三要素格式

每条反馈记忆必须包含以下三部分：

```markdown
## YYYY-MM-DD — [feedback] [scope:个人|项目]
- 规则：集测不要 mock 数据库连接
- 原因：mock 和生产环境行为不一致，上次导致迁移事故
- 适用场景：集测不该 mock；纯逻辑单元测试可以 mock
```

### scope 字段说明

- `[scope:个人]` — 只对先生有效的个人偏好，如"不要再回复末尾加总结"
- `[scope:项目]` — 项目规范，如"集测必须用真实数据库"

## 三、5 类否决清单

前置过滤器的第一道关卡。遇以下 5 类信息**直接跳过**，不进任何记忆层：

| # | 禁区 | 原因 | 替代方案 |
|---|------|------|---------|
| 1 | ❌ 代码结构/架构/文件路径 | 可从源代码实时读取 | 用 read/exec 查当前状态 |
| 2 | ❌ git 提交历史 | git 是权威来源 | 用 exec git log |
| 3 | ❌ 调试修复过程 | 修复已在代码里 | 代码是最好的参考 |
| 4 | ❌ 文档/配置已写过的内容 | 重复存储制造混乱 | 引回原始文档路径 |
| 5 | ❌ 临时任务细节/中间状态 | 短期上下文 | 用 proactivity 文件（会话内） |

**特殊规则：** 即使用户明确要求记，如果属于上述 5 类，反问："这部分我能直接从代码/工具查到，你真正想我记住的是哪个洞察？"

## 四、前置过滤器完整判断流程

```
收到一条信息
  │
  ├─ Step 1: 否决清单 → 命中 → ⛔ 跳过
  │
  ├─ Step 2: 洞察检查
  │   ├─ 这是洞察（可复用的结论/判断/原因）？→ 继续
  │   └─ 这是活动日志（单纯记录了发生过什么）？→ ⛔ 跳过
  │
  ├─ Step 3: 触发条件 T1-T7
  │   T1  先生明确要求"记住"
  │   T2  涉及已知人物
  │   T3  做出决策/选择
  │   T4  表达了个人偏好
  │   T5  有明确时间点的待办
  │   T6  踩坑/教训/纠正
  │   T7  直觉（不用纠结理由）
  │   └─ 全部未命中 → 暂不处理，等第一层补漏
  │
  ├─ Step 4: 类型识别
  │   ├─ 角色/偏好/背景 → 🧑 用户记忆 → MEMORY.md
  │   ├─ 纠正/肯定 → 💬 反馈记忆 → corrections.md（三要素格式）
  │   ├─ 项目动机/时间线 → 📋 项目记忆 → projects/<name>.md
  │   └─ 外部资源在哪 → 🔗 引用记忆 → 就近存放
  │
  └─ Step 5: 写入前冲突检测
      ├─ 已存在且一致 → 跳过
      ├─ 已存在但需更新 → 旧条目标"过时"+写新
      ├─ 已存在但需纠错 → 直接覆盖
      ├─ 部分重叠 → 补充元数据，不新增
      └─ 不确定是否值得写 → 暂存池
```

## 五、写入机制

### 5.1 谁来写

两条路径，互斥执行：

```
主模型（对话中写入）
    └─ 先生明确要求"记住 XX" → 主模型即时写入
    └─ 写入后标记此对话已处理记忆

会话后提取（对话结束时自动）
    └─ 触发条件：主模型不再调用工具
    └─ 互斥检查：本次对话主模型已写过记忆？→ 跳过
    └─ 权限：只读（读对话历史+读现有记忆文件）+ 写（记忆目录）
    └─ 禁止：修改代码、执行副作用命令、调外部服务
    └─ 轮次限制：最多 5 轮对话完成（理想：2 轮）
        ├─ 第 1 轮：并行读现有记忆（去重）+ 读本次新对话
        └─ 第 2 轮：并行写入提炼结果
    └─ 成功 → 游标推进；失败 → 游标不推进
    └─ 节流：可配置间隔，高频场景每隔 N 轮触发一次
```

### 5.2 写入格式

每条记忆的标准格式：

```markdown
### YYYY-MM-DD [type] [scope] [tag1 tag2]
- 摘要：一句话描述（供检索用）
- 详情：具体内容
  - 背景/原因
  - 关键细节
```

### 5.3 游标机制

```
cursor.json
{
  "lastProcessedMessageId": "xxx",
  "lastExtractTime": "2026-07-07T19:00:00+08:00",
  "success": true
}
```

- 每次会话后提取成功 → 更新游标
- 提取失败 → 游标不动，下次重新处理
- 合并：如果上次提取还在跑、新对话又结束 → 不并行，等上次完成后再用最新上下文跑

## 六、检索机制（Active Recall）

每轮对话开始时执行：

```
  ┌─────────────────────────────────────────────────┐
  │                 用户输入                          │
  └──────────────────┬──────────────────────────────┘
                     │
  ┌──────────────────▼──────────────────────────────┐
  │ Phase 1: 扫描                                    │
  │ · 递归扫描记忆目录                               │
  │ · 上限 200 个文件                               │
  │ · 每个文件只读前 30 行（front matter 元数据）    │
  │ · 成本：纯文件 I/O，零模型调用                    │
  └──────────────────┬──────────────────────────────┘
                     │
  ┌──────────────────▼──────────────────────────────┐
  │ Phase 2: 摘要                                    │
  │ · 将所有元数据拼成一行一条的清单                  │
  │ · 格式：[类型] [标签] [时间] 一句话描述          │
  │ · 去除已在本对话注入过的文件（去重）              │
  │ · 成本：几千 token                               │
  └──────────────────┬──────────────────────────────┘
                     │
  ┌──────────────────▼──────────────────────────────┐
  │ Phase 3: 选择                                    │
  │ · 把用户输入 + 记忆清单发给 DeepSeek V4 Flash    │
  │ · 提示词：宁缺毋滥，不确定不选，最多选 5 条      │
  │ · 工具文档类（正在顺利使用的）不选               │
  │ · 工具的「已知问题/注意事项」要选                │
  └──────────────────┬──────────────────────────────┘
                     │
  ┌──────────────────▼──────────────────────────────┐
  │ Phase 4: 后校验                                  │
  │ · 确定性代码检查：所有选中的文件名是否真实存在    │
  │ · 防止模型幻觉出不存在文件                        │
  └──────────────────┬──────────────────────────────┘
                     │
  ┌──────────────────▼──────────────────────────────┐
  │ Phase 5: 注入                                    │
  │ · 读取选中文件的完整内容                         │
  │ · 上限 ≤4KB/条，超长截断 + 附指针提示            │
  │ · 每轮 ≤5 条，单轮注入 ≤20KB                     │
  │ · 整个会话累计 ≤60KB（防上下文撑爆）             │
  │ · 每条附带时间标签：                              │
  │   - 今天/昨天 → 不加警告                        │
  │   - ≥2天 → 附加过期提醒                         │
  └─────────────────────────────────────────────────┘
```

### 过期警告模板

```
⚠️ 这条记忆是 N 天前记录的，是当时的时间点快照，
不是当前实时状态。其中关于代码行为或文件位置的描述可能已过时，
在基于此做决策前请先验证当前代码的实际状态。
```

## 七、跨系统路由表

| 记什么 | 写到哪里 | 时效 | 检索优先级 |
|--------|---------|------|-----------|
| 事实/事件/日常记录 | `memory/YYYY-MM-DD.md` | 长期（raw log） | 低（不主动检索） |
| 🧑 用户记忆（偏好/决策/人物） | `MEMORY.md` | 长期 | 🔴 最高 |
| 💬 反馈记忆（纠正/肯定） | `~/self-improving/corrections.md` | 长期 | 🔴 最高 |
| 可复用规则/偏好 | `~/self-improving/memory.md` | 长期 | 🟡 高 |
| 📋 项目记忆 | `~/self-improving/projects/<name>.md` | 项目周期 | 🟡 项目激活时高 |
| 🔗 引用记忆 | 就近存放 | 长期 | 🟡 中 |
| 领域知识 | `~/self-improving/domains/<domain>.md` | 长期 | 🟢 按需 |
| 待确认条目 | `memory/pending-memory.md` | 7天过期 | ⚪ 不参与检索（兜底） |
| 任务状态/阻塞/下一步 | `~/proactivity/session-state.md` | 会话级 | ⚪ 不参与检索 |
| 过程面包屑 | `~/proactivity/memory/working-buffer.md` | 会话级 | ⚪ 不参与检索 |

## 八、上下文加载策略（每次任务前）

```
固定预加载（无条件）：
  ├── MEMORY.md（长期用户信息）
  └── ~/self-improving/memory.md（可复用规则）

Active Recall 动态注入：
  ├── 扫描 → 摘要 → 选择(≤5条) → 后校验 → 注入
  ├── 每条带时间标签 + 过期警告
  └── 上限：单轮 ≤20KB / 整对话 ≤60KB

不加载（用工具实时查）：
  ├── 代码文件 → read
  ├── git 历史 → exec git
  └── 文档配置 → read
```

## 附录：记忆文件 front matter 格式（供 Active Recall 扫描用）

所有记忆文件的推荐头部格式，确保前 30 行包含足够的检索元数据：

```markdown
---
name: 记忆条目名称
type: user | feedback | project | reference
scope: personal | project
tags: [tag1, tag2]
date: 2026-07-07
summary: 一句话描述（供 Active Recall 摘要用）
---

内容正文...
```

当前旧文件无需强制改造，新写入时按此格式即可。
